Q1 What is React? Why do we use it? +
CoreBasics

React is an open-source JavaScript library created by Meta (Facebook) for building user interfaces. It focuses entirely on the View layer — what the user sees and interacts with. It is not a framework; it doesn't dictate how you handle routing, HTTP calls, or styling. You pick and combine tools you like.

Problem without ReactHow React solves it
Manually updating DOM is slow and error-proneVirtual DOM handles updates efficiently
Code gets messy in large appsComponent-based — break UI into small reusable pieces
Hard to share UI logicProps + custom hooks allow reuse across components
Hard to manage app stateuseState, useReducer, Context built-in
// Without React — imperative (tell browser HOW to update)
document.getElementById('count').innerText = count;
document.getElementById('btn').addEventListener('click', () => { ... });

// With React — declarative (describe WHAT you want)
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
Interview tip
React is a library, not a framework. The key distinction: with a library, you are in control and call it. With a framework like Angular, it calls your code. Next.js is a framework built on top of React.
Q2 What is the difference between a Library and a Framework? +
Basics

The core difference is about who is in control — known as the "Inversion of Control" principle.

Library (React)Framework (Angular)
ControlYou call the library when neededFramework calls your code at designated points
FlexibilityPick any tools alongside itFollows framework conventions and structure
Size/ScopeSmall — just the view layerFull package: routing, HTTP, forms, etc.
Learning curveEasier to start, compose yourselfSteeper but more structured and opinionated
Real-world analogy
A library is like a toolbox — you pick the hammer, the screwdriver, whatever you need. A framework is like a pre-built construction system — everything is already decided for you, you just follow the workflow.
Q3 What is JSX? +
CoreJSX

JSX stands for JavaScript XML. It's a syntax extension that lets you write HTML-like code directly inside JavaScript. Babel (a transpiler) converts it into regular JavaScript before it runs in the browser.

JSX is not mandatory — you could write React.createElement() calls manually — but JSX makes code far more readable and is used in virtually every React codebase.

// JSX (what you write)
const element = <h1 className="title">Hello World</h1>;

// What Babel compiles it to
const element = React.createElement('h1', { className: 'title' }, 'Hello World');

Key JSX rules to remember:

  • Must return one root element — wrap siblings in <></> (Fragment) if needed
  • Use className instead of class (since class is a JS keyword)
  • All tags must be closed — even self-closing ones: <img />, <br />
  • JavaScript expressions go inside { } curly braces
  • Attributes use camelCase: onClick, onChange, maxLength
Common gotcha
JSX is not HTML. class → className, for → htmlFor, and event names are camelCase (onclick → onClick). These trip up many beginners.
Q4 What is a Component in React? +
CoreComponents

A component is an independent, reusable piece of UI. Think of it like a LEGO brick — you build complex interfaces by combining small, focused components. Each component manages its own structure, style, and behavior.

React has two types: Functional components (modern, use these always) and Class components (legacy, avoid in new code). Since React 16.8 introduced Hooks, functional components can do everything class components can.

// Functional Component — modern standard
function Welcome({ name }) {
  return <h1>Hello, {name}!</h1>;
}

// Use it like an HTML element
<Welcome name="Prathamesh" />
// Renders: Hello, Prathamesh!
Naming rule
Component names must start with a capital letter. <button> is a native HTML element; <Button> is your custom React component. This is how React tells them apart.
Q5 What are Props? +
CoreProps

Props (short for Properties) are the mechanism for passing data from a parent component to a child component. They flow in one direction only: top-down. A child component receives props as its function arguments.

Props are read-only — a child should never try to modify its own props. This one-way data flow makes React apps predictable and easy to debug. If you need to update data, it must happen in the parent.

function App() {
  return (
    <UserCard
      name="Prathamesh"
      age={22}
      isAdmin={true}
      onClick={() => alert('clicked')}
    />
  );
}

function UserCard({ name, age, isAdmin, onClick }) {
  return (
    <div onClick={onClick}>
      <h2>{name}</h2>
      <p>Age: {age}</p>
      {isAdmin && <span>Admin</span>}
    </div>
  );
}
Default props
You can set default values directly in the function signature: function Button({ label = 'Click me', color = 'blue' }). If the parent doesn't pass that prop, the default is used.
Q6 What is State? +
StateuseState

State is a component's own internal data that can change over time. Unlike props (which come from the parent), state is created and owned by the component itself. Crucially, when state changes, React automatically re-renders the component to reflect the new data in the UI.

In functional components, state is managed using the useState hook. It returns a pair: the current value and a setter function. You must always use the setter — never mutate state directly.

import { useState } from 'react';

function Counter() {
  const [count, setCount] = useState(0); // initial value = 0

  return (
    <div>
      <p>Count: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(count - 1)}>-</button>
      <button onClick={() => setCount(0)}>Reset</button>
    </div>
  );
}
Never mutate state directly
count = count + 1 ❌ — this changes the variable but doesn't trigger a re-render.
setCount(count + 1) ✅ — this tells React state changed, triggering a re-render.

A simple mental model: props are like function arguments — passed in from outside. State is like local variables — defined and managed inside the function.

Key insight
The same piece of data can be state in one component and a prop in another. A parent stores it as state (can change it), then passes it as a prop to children (who read it but can't change it).
Q8 What is the Virtual DOM? +
InternalsPerformance

The Virtual DOM is a lightweight JavaScript object representation of the real DOM, kept in memory by React. Instead of updating the real DOM every time something changes, React first updates its virtual copy, compares old vs. new (called diffing), and then makes the minimum necessary changes to the real DOM.

// The update flow:
// 1. State changes → React re-renders component (creates new Virtual DOM)
// 2. React DIFFS: new Virtual DOM vs old Virtual DOM
// 3. React calculates minimal set of real DOM updates needed
// 4. React updates only those parts of the real DOM
// 5. Browser paints the minimal changes

Real DOM operations (adding, removing, modifying elements) are expensive because they trigger browser reflows and repaints. Comparing plain JavaScript objects (virtual DOM) is extremely fast by comparison. React also batches multiple updates into a single DOM pass.

Misconception
The Virtual DOM isn't universally faster — for very simple updates, it adds overhead. Its real benefit is providing a declarative programming model (you describe what the UI should look like, React figures out how to get there) while being fast enough for complex UIs.
Q9 What is the difference between React.createElement and JSX? +
JSXInternals

They produce exactly the same result. JSX is just syntactic sugar — Babel transforms every JSX element into a React.createElement() call at build time. In React 17+, the import is automatic, so you no longer need import React from 'react' just to use JSX.

// JSX (what you write)
const element = (
  <div className="card">
    <h1>Hello</h1>
  </div>
);

// What Babel compiles it to (React 16 and below)
const element = React.createElement(
  'div',
  { className: 'card' },
  React.createElement('h1', null, 'Hello')
);

// Both produce the same React element object:
// { type: 'div', props: { className: 'card', children: [...] } }
Q10 How do you render a list in React? +
ListsKeys

Use JavaScript's .map() method to transform an array of data into an array of JSX elements. React renders arrays of elements natively. Always provide a unique key prop for each item — React uses it for efficient updates.

const users = [
  { id: 1, name: 'Prathamesh' },
  { id: 2, name: 'Rahul' },
  { id: 3, name: 'Priya' },
];

function UserList() {
  return (
    <ul>
      {users.map(user => (
        <li key={user.id}>{user.name}</li>
      ))}
    </ul>
  );
}
Q11 What is the key prop and why is it needed? +
ListsReconciliation

The key prop helps React identify which specific list items changed, were added, or were removed between renders. Without keys, React has to re-render every item in the list any time one item changes.

// ❌ Using index — causes bugs when list order changes
{todos.map((todo, i) => <TodoItem key={i} todo={todo} />)}

// ✅ Using stable unique ID — always correct
{todos.map(todo => <TodoItem key={todo.id} todo={todo} />)}

Why index-as-key is harmful: When you delete an item at position 0, all remaining items shift — their index-based keys change. React sees key=0 as "changed" (not deleted), so it updates 2 items instead of removing 1. With ID-based keys, React correctly removes just the deleted item's DOM node.

Key rules
Keys must be unique among siblings (not globally). Keys must be stable — using Math.random() as a key is even worse than index because it generates a new key every render, causing every component to remount.
Q12 What is Conditional Rendering? +
JSXPatterns

Conditional rendering means showing or hiding UI elements based on a condition — like showing a login button only when the user is logged out. React uses standard JavaScript conditionals since JSX compiles to JS.

function UserGreeting({ isLoggedIn, name }) {
  // Method 1: if/else — good for large blocks
  if (!isLoggedIn) return <p>Please log in</p>;

  return (
    <div>
      {/* Method 2: Ternary — great for either/or */}
      <h1>{isLoggedIn ? `Welcome, ${name}!` : 'Stranger'}</h1>

      {/* Method 3: && — show only when true */}
      {isLoggedIn && <button>Logout</button>}

      {/* Method 4: Nullish fallback */}
      <p>{name || 'Anonymous'}</p>
    </div>
  );
}
Classic gotcha — the 0 bug
{count && <Badge />} renders "0" when count is 0 because 0 is a falsy value that JSX renders as text.
Fix: {count > 0 && <Badge />} — use an explicit comparison that returns a boolean.
Q13 How do you handle events in React? +
EventsHandlers

React uses synthetic events — wrappers around native browser events that normalize behavior across browsers. Event names are written in camelCase (onClick, not onclick), and you pass a function reference, not a string.

function Form() {
  const handleSubmit = (e) => {
    e.preventDefault(); // prevent page reload
    console.log('submitted');
  };

  const handleChange = (e) => {
    console.log(e.target.value); // get input value
  };

  return (
    <form onSubmit={handleSubmit}>
      <input onChange={handleChange} />
      <button type="submit">Submit</button>
    </form>
  );
}

// Passing arguments — wrap in arrow function
<button onClick={() => handleDelete(item.id)}>Delete</button>
// ❌ Wrong: onClick={handleDelete(item.id)} — calls immediately on render!
Q14 What is a Fragment in React? +
JSX

A Fragment lets you return multiple sibling elements without adding an extra wrapper node to the DOM. This matters because extra wrappers can break CSS layouts (flexbox/grid parent rules), create invalid HTML (e.g., table rows), or add unnecessary nesting.

// ❌ Extra div pollutes DOM and can break flex/grid layouts
return (<div><h1>Title</h1><p>Content</p></div>);

// ✅ Short syntax Fragment — renders nothing in DOM
return (<><h1>Title</h1><p>Content</p></>);

// ✅ Explicit syntax — use when you need a key prop (in lists)
return (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.description}</dd>
  </React.Fragment>
);
Q15 What is useState? +
HooksState

useState is the foundational React hook for adding local state to functional components. It returns a tuple: [currentValue, setterFunction]. The setter can accept either a new value directly, or a function that receives the previous state (the functional update form — use this when the new state depends on the old one, especially in async contexts).

const [count, setCount] = useState(0);

// Direct update
setCount(5);

// Functional update — safe in async / closures
setCount(prev => prev + 1);

// Updating object state — always spread old state first!
const [user, setUser] = useState({ name: '', age: 0 });
setUser(prev => ({ ...prev, name: 'Prathamesh' })); // keeps age

// Lazy initialization — pass a function for expensive initial value
const [data, setData] = useState(() => parseExpensiveData());
// parseExpensiveData only runs once, not on every render
State updates are asynchronous
Calling setCount(count + 1) three times in a row won't give you +3 — all three calls see the same stale count. Use the functional form setCount(c => c + 1) to chain correctly.
Q16 What is useEffect? +
HooksSide Effects

useEffect lets you perform side effects in functional components — anything that reaches outside React's render cycle: API calls, timers, DOM manipulation, event listener setup, localStorage reads, etc.

It accepts two arguments: a callback function (the effect) and an optional dependency array that controls when the effect re-runs. Optionally, the callback can return a cleanup function that runs before the next effect execution or on component unmount.

// 1. Runs after EVERY render (no array) — rarely needed
useEffect(() => { document.title = 'Updated'; });

// 2. Runs ONCE after mount (empty array) — like componentDidMount
useEffect(() => { fetchInitialData(); }, []);

// 3. Runs when dependency changes
useEffect(() => {
  fetchUser(userId);
}, [userId]);

// 4. With cleanup — prevents memory leaks
useEffect(() => {
  const timer = setInterval(() => setCount(c => c + 1), 1000);
  return () => clearInterval(timer); // runs on unmount!
}, []);
The isMounted pattern
In async effects, always guard state updates with an isMounted flag or AbortController to prevent "can't perform state update on unmounted component" warnings when a component unmounts before a fetch completes.
Q17 What is useRef? +
HooksDOM

useRef returns a mutable object { current: value } that persists across renders. The key difference from state: changing .current does NOT trigger a re-render. This makes it perfect for two use cases: accessing DOM nodes directly, and storing values that need to persist but shouldn't cause re-renders (like timer IDs, previous values, or flags).

// Use case 1: DOM access
function TextInput() {
  const inputRef = useRef(null);
  return (
    <>
      <input ref={inputRef} />
      <button onClick={() => inputRef.current.focus()}>Focus</button>
    </>
  );
}

// Use case 2: Persist value without re-render
function Timer() {
  const intervalRef = useRef(null);
  const start = () => {
    intervalRef.current = setInterval(() => console.log('tick'), 1000);
  };
  const stop = () => clearInterval(intervalRef.current);
}
Ref vs State
Use state when the value should be displayed in the UI (needs re-render). Use ref when you need to track something internally but the UI doesn't depend on it directly (e.g., tracking whether a component mounted, storing the previous value).
Q18 What is useContext? +
HooksContext

useContext lets any component in a tree read from a Context without prop drilling. Context provides a way to share data globally across a component tree — like the current theme, authenticated user, or locale. Three steps: create → provide → consume.

// 1. Create context (with a default value)
const ThemeContext = createContext('light');

// 2. Wrap your component tree in the Provider
function App() {
  const [theme, setTheme] = useState('light');
  return (
    <ThemeContext.Provider value={{ theme, setTheme }}>
      <Header /> // can access theme without props!
      <Main />
    </ThemeContext.Provider>
  );
}

// 3. Read it anywhere in the tree — no prop threading needed
function Header() {
  const { theme, setTheme } = useContext(ThemeContext);
  return <header className={theme}>...</header>;
}
Q19 What is Prop Drilling and why is it a problem? +
PatternsContext

Prop drilling is when you pass data through multiple intermediate component layers that don't actually use the data — just to get it to a deeply nested child that needs it. It makes code harder to maintain: if the data shape changes, you must update every component in the chain.

// ❌ Prop drilling — Layout and Sidebar don't use 'user'
App({ user }) → <Layout user={user}/> → <Sidebar user={user}/> → <UserAvatar user={user}/>

// ✅ Context — UserAvatar reads directly, no drilling
function UserAvatar() {
  const { user } = useContext(UserContext); // reads directly!
  return <img alt={user.name} />;
}
When prop drilling is OK
1-2 levels of passing props is completely fine — don't over-engineer with Context for shallow trees. Context adds indirection and makes components harder to reuse in isolation. Use it for truly global data (auth, theme, locale) or data shared across distant parts of the tree.
Q20 What is the difference between null, undefined, and false in JSX? +
JSXGotchas

null, undefined, false, and true are all valid JSX children that render nothing. This is intentional — it enables conditional rendering patterns like {isLoggedIn && <Menu />}.

{null}       // renders nothing
{undefined}  // renders nothing
{false}      // renders nothing
{true}       // renders nothing
{0}          // ⚠️ renders "0" — common bug!
{''}         // renders empty string (invisible)

// Classic 0 bug — always use boolean condition with &&
const count = 0;
{count && <Badge />}      // ❌ renders "0" on screen!
{count > 0 && <Badge />}  // ✅ renders nothing
{!!count && <Badge />}    // ✅ also works (double negation)
Q21 How do you pass data from Child to Parent? +
PatternsCallbacks

React's data flow is unidirectional — parent to child only. To send data upward, the parent passes a callback function as a prop. The child calls that function with the data it wants to send up. This pattern preserves React's one-way data flow.

function Parent() {
  const [msg, setMsg] = useState('');
  return (
    <div>
      <p>Child said: {msg}</p>
      <Child onSend={setMsg} /> // pass setter as callback
    </div>
  );
}

function Child({ onSend }) {
  const [input, setInput] = useState('');
  return (
    <>
      <input value={input} onChange={e => setInput(e.target.value)} />
      <button onClick={() => onSend(input)}>Send</button>
    </>
  );
}
Q22 What are Controlled and Uncontrolled Components? +
FormsPatterns

A controlled component is an input whose value is driven by React state — React is the "single source of truth." An uncontrolled component lets the DOM manage its own state; you read it when needed via a ref.

ControlledUncontrolled
Source of truthReact stateDOM itself
Access valuevalue prop + onChangeref.current.value
Best forValidation, formatting, dependent fieldsFile inputs, simple forms
// Controlled — React owns the value
const [val, setVal] = useState('');
<input value={val} onChange={e => setVal(e.target.value)} />

// Uncontrolled — DOM owns the value, read on demand
const ref = useRef();
<input defaultValue="initial" ref={ref} />
// Read on submit: ref.current.value
Q23 What is React.StrictMode? +
Dev ToolsDebugging

React.StrictMode is a development-only wrapper that activates extra checks and warnings to help you write better React code. It has no effect in production.

Most notably in React 18, it intentionally mounts every component twice (mount → unmount → mount) to surface bugs from missing cleanup functions in effects. If your app breaks on double-mount, you have a cleanup problem.

ReactDOM.createRoot(document.getElementById('root')).render(
  <React.StrictMode>
    <App />
  </React.StrictMode>
);

What StrictMode detects: deprecated API usage, missing cleanup in effects, impure render functions, legacy context usage, and deprecated findDOMNode.

Q24 What is the difference between export default and named export? +
JavaScriptModules
// Named export — exact name required when importing
export function Button() { ... }
export const API_URL = 'https://api.example.com';
import { Button, API_URL } from './components'; // must match

// Default export — can import with any name
export default function App() { ... }
import App from './App';

// A file can have ONE default + MANY named exports
Convention
Most React projects use default exports for page/component files, and named exports for utility functions and constants. Consistency matters more than which you pick.
Q25 What is the children prop? +
PropsComposition

children is a special built-in prop that contains whatever is placed between a component's opening and closing tags. It's the foundation of React's composition model — building "container" components that wrap other content.

function Card({ title, children }) {

      <div className="card-body">{children}</div>

}

// Everything between the tags becomes children

Q26 Explain all useEffect dependency array cases +
HooksuseEffect

The dependency array is the most nuanced part of useEffect. It controls when the effect runs — not just whether it runs. ESLint's exhaustive-deps rule will tell you if you're missing dependencies.

// No array — runs after EVERY render. Usually a bug.
useEffect(() => { console.log('every render'); });

// Empty array — runs ONCE after mount (like componentDidMount)
useEffect(() => { setupListeners(); return () => cleanup(); }, []);

// With dependencies — runs on mount AND when dep changes
useEffect(() => { fetchUser(userId); }, [userId]);

// ⚠️ Object dependency — always a new reference = always runs!
const options = { page: 1 }; // new object every render
useEffect(() => { fetchData(options); }, [options]); // ❌ infinite loop risk

// ✅ Use primitive value OR useMemo for objects
useEffect(() => { fetchData({ page }); }, [page]); // only runs when page changes
Missing dependencies trap
If you read a prop or state variable inside useEffect but don't add it to the deps array, the effect will run with a stale closure — seeing old values. Always follow the exhaustive-deps ESLint rule. If adding a dep causes infinite loops, the real fix is usually useMemo or useCallback.
Q27 What is useMemo? +
HooksPerformance

useMemo caches the result of an expensive calculation between renders. It only recalculates when one of its dependencies changes. Think of it as memoization for values.

// Without useMemo — filters on EVERY render
const filtered = products.filter(p =>
  p.name.toLowerCase().includes(query.toLowerCase())
);

// With useMemo — only recalculates when products or query changes
const filteredProducts = useMemo(() =>
  products.filter(p => p.name.toLowerCase().includes(query)),
  [products, query]
);
When to NOT use useMemo
For simple operations (string concatenation, basic math), useMemo overhead (dependency comparison) often costs more than just recomputing. Profile first. Use it for: complex array filtering/sorting, expensive object creation, values passed as deps to other hooks or to memoized children.
Q28 What is useCallback? +
HooksPerformance

useCallback returns a memoized function reference — the same function object between renders unless its dependencies change. Without it, a new function is created every render, which breaks React.memo on child components (since they'd see a "new" prop every time).

// ❌ Without useCallback — new function every render
// → memo on child is useless, child always re-renders
const handleDelete = (id) => setTodos(prev => prev.filter(t => t.id !== id));

// ✅ With useCallback — stable reference
// → memo on child works, only re-renders when todo data changes
const handleDelete = useCallback((id) => {
  setTodos(prev => prev.filter(t => t.id !== id));
}, []); // empty array — setTodos is stable
useMemo vs useCallback
They solve the same problem for different things. useMemo caches a computed value. useCallback caches a function. In fact, useCallback(fn, deps) is equivalent to useMemo(() => fn, deps).
Q29 What is useReducer? +
HooksState Management

useReducer is an alternative to useState for managing complex state logic with multiple sub-values or state transitions that depend on the previous state. The pattern: all state transitions are described as "actions" dispatched to a pure reducer function.

function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD':    return { ...state, items: [...state.items, action.payload] };
    case 'REMOVE': return { ...state, items: state.items.filter(i => i.id !== action.payload) };
    case 'CLEAR':  return { items: [], total: 0 };
    default:      return state;
  }
}

const [state, dispatch] = useReducer(cartReducer, { items: [], total: 0 });

// Dispatch actions instead of calling setters
dispatch({ type: 'ADD', payload: { id: 1, name: 'Shirt' } });
dispatch({ type: 'CLEAR' });
useState vs useReducer
Use useState when: simple primitive values, independent fields. Use useReducer when: multiple related values that change together, next state depends on previous, complex transitions, or you want Redux-like testability (reducers are pure functions — easy to test in isolation).
Q30 What are Custom Hooks? +
HooksPatterns

A custom hook is a regular JavaScript function whose name starts with use and that calls one or more React hooks. It's the primary mechanism for extracting and reusing stateful logic across multiple components — without changing their UI or component hierarchy.

// Custom hook — encapsulates fetch logic
function useFetch(url) {
  const [data, setData]       = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    let mounted = true;
    fetch(url)
      .then(r => r.json())
      .then(d => { if (mounted) setData(d); })
      .catch(e => { if (mounted) setError(e.message); })
      .finally(() => { if (mounted) setLoading(false); });
    return () => { mounted = false; };
  }, [url]);

  return { data, loading, error };
}

// Now reuse in any component cleanly
const { data: user, loading } = useFetch(`/api/users/${id}`);
Good custom hook candidates
Data fetching, form state + validation, debouncing, local storage sync, window resize/scroll tracking, websocket connection, infinite scroll, media query matching. Anything you find yourself copy-pasting between components is a custom hook waiting to be born.
Q31 Rules of Hooks — what are they and why do they exist? +
HooksRules

There are two rules: (1) Only call hooks at the top level of a function — never inside conditionals, loops, or nested functions. (2) Only call hooks from React function components or custom hooks — not regular JS functions.

// ❌ WRONG — hook inside condition (breaks call order)
if (isLoggedIn) { const [data] = useState(null); }

// ❌ WRONG — hook inside loop
for (let i = 0; i < count; i++) { const [val] = useState(i); }

// ✅ Correct — hooks at top level, conditions go inside
const [data, setData] = useState(null);
useEffect(() => { if (isLoggedIn) fetchData(); }, [isLoggedIn]);

Why this rule? React tracks hooks internally by their call order in an array. Every render must call the same hooks in the same order. If a conditional skips a hook, the array index shifts and React maps the wrong state to the wrong hook — corrupting all subsequent state.

Q32 What is React.memo? +
PerformanceMemoization

React.memo is a Higher Order Component that wraps your component and skips re-rendering if the props haven't changed (shallow comparison by default). Without it, a child re-renders any time its parent re-renders — even if the child's props are identical.

import { memo } from 'react';

// Wrapped with memo — skips re-render if name + email unchanged
const UserCard = memo(function({ name, email }) {
  return <div>{name} — {email}</div>;
});

// Custom comparison (optional) — for complex objects
const UserCard = memo(
  ({ user }) => <div>{user.name}</div>,
  (prev, next) => prev.user.id === next.user.id // true = skip
);
memo only works with stable props
memo is useless if the parent passes a new object literal or arrow function every render (e.g. <Child style={{ color: 'red' }} />). Always pair with useMemo and useCallback for object/function props.
Q33 What is the difference between useEffect and useLayoutEffect? +
HooksDOM

Both run after React updates the DOM. The timing difference: useLayoutEffect runs synchronously before the browser has a chance to paint, while useEffect runs asynchronously after the paint.

// Timeline:
// React updates DOM
// → useLayoutEffect fires (sync) ← DOM measurements here
// → Browser paints (user sees change)
// → useEffect fires (async)

// Use useLayoutEffect when measuring DOM before paint

Q35 What are Error Boundaries? +

Error Boundaries are class components that catch JavaScript errors in their child component tree and display a fallback UI instead of crashing the entire app. Without them, a single component error unmounts the entire React tree.

  state = { hasError: false };

  static getDerivedStateFromError() { return { hasError: true }; }

  componentDidCatch(error, info) { logErrorToService(error, info); }

  render() {
    if (this.state.hasError) return <h1>Something went wrong.</h1>;
    return this.props.children;
  }
}

Q46 What is Reconciliation in React? +
InternalsPerformance

Reconciliation is React's algorithm for efficiently computing the minimum set of DOM changes needed when the virtual DOM changes. React follows three key heuristics:

  1. Different types = full rebuild. If a <div> becomes a <span>, React destroys the subtree and builds fresh (all child state is lost).
  2. Same type = update attributes only. If an element type stays the same, React patches only what changed.
  3. Lists use keys. Keys let React match items between old and new lists — enabling moves, insertions, and deletions without re-rendering untouched items.
Type-change gotcha
Conditionally rendering a different component type at the same position resets all child state. If you need to preserve state, keep the same component type and conditionally change its content/props instead.

Q48 What is the React Fiber architecture? +
InternalsFiber

Fiber is React's complete rewrite of its internal reconciliation engine, shipped in React 16. Before Fiber, React's reconciliation was recursive and synchronous — once started, it couldn't be interrupted. A massive tree update would block the main thread for hundreds of milliseconds, making UIs janky.

Fiber reimplements the algorithm as a linked list of "units of work" (one Fiber node per component). This data structure allows React to pause after processing each unit, yield control back to the browser, and resume later — making reconciliation incremental and interruptible.

// What Fiber enables:
// 1. Pause reconciliation → browser handles urgent input
// 2. Resume from where it stopped
// 3. Abort work that became stale (user typed again)
// 4. Assign priority levels to different updates
//    - User input → HIGH priority (immediate)
//    - Data fetch result → LOW priority (can defer)
//
// This is the foundation for: Suspense, useTransition,
// Concurrent rendering, Server Components
Two phases
Fiber splits work into the render phase (interruptible — builds a "work-in-progress" fiber tree, diffs old vs new) and the commit phase (non-interruptible — applies the minimal DOM mutations, runs effects). Only the commit phase touches the real DOM.
Q49 What is Concurrent Mode / Concurrent Rendering? +
React 18Concurrent

Concurrent rendering (React 18) is the ability to prepare multiple versions of the UI simultaneously in the background without blocking the browser's main thread. React can start rendering an update, pause if a more urgent update arrives, handle the urgent one first, then return to the background work.

// Opt in with createRoot (React 18)
const root = ReactDOM.createRoot(document.getElementById('root'));
root.render(<App />);

// React can now:
// - Interrupt a low-priority render (search results) for
//   a high-priority render (keypress, click)
// - Reuse rendered work if nothing changed
// - Throw away stale renders (user typed new query)
// - Show consistent UI snapshots — no tearing
Opt-out
Only createRoot enables concurrent features. If you use the old ReactDOM.render(), you get legacy synchronous rendering — no concurrent features. All third-party library updates for React 18 involve ensuring compatibility with the new concurrent model.

Q54 What is Automatic Batching in React 18? +
React 18Performance

Batching means React groups multiple state updates together into a single re-render. In React 17, batching only happened inside React event handlers. In React 18 with createRoot, all state updates are batched automatically — including inside setTimeout, Promises, and native event handlers.

// React 17 — NOT batched in async context (2 renders)
setTimeout(() => {
  setCount(c => c + 1); // render 1
  setFlag(f => !f);      // render 2
}, 1000);

// React 18 — batched everywhere (1 render) ✅
setTimeout(() => {
  setCount(c => c + 1);
  setFlag(f => !f);      // only 1 render total
}, 1000);

// Opt out when needed (force immediate render)
import { flushSync } from 'react-dom';
flushSync(() => setCount(c => c + 1)); // immediate render
flushSync(() => setFlag(f => !f));      // immediate render

The React Compiler is a build-time Babel plugin that automatically inserts memoization — eliminating the need to write useMemo, useCallback, and React.memo manually. It statically analyzes your component's data flow and inserts the right optimizations at compile time.

function Component({ items, filter }) {
  const sorted = useMemo(() => items.filter(i => i.type === filter).sort(), [items, filter]);

}

// 1. TRIGGER — what starts a render
//    - setState / dispatch
//    - Parent re-renders (new props)

//    React calls your component function

By design — effects can trigger more state updates, which would cause more renders. Running them after paint means the user already sees a consistent UI before any cascading updates happen. This avoids the "flash" of intermediate states.

Q60 What is the difference between useEffect and server-side code? +
SSRuseEffect

useEffect never runs on the server — it's a client-only hook. This is by design: servers don't have browser APIs (localStorage, window, document). Effects are skipped during SSR, which means any browser-specific code must live inside an effect (or check for window).

// ✅ Safe — browser APIs inside useEffect (client-only)
useEffect(() => {
  localStorage.setItem('key', value);
  window.scrollTo(0, 0);
  document.title = 'Updated';
}, [value]);

// ❌ Server crash — localStorage undefined on server!
function Component() {
  const theme = localStorage.getItem('theme'); // crashes SSR
  return <div className={theme}>...</div>;
}

// ✅ SSR-safe — read in effect, show fallback during SSR
const [theme, setTheme] = useState('light'); // safe default for SSR
useEffect(() => {
  const saved = localStorage.getItem('theme');
  if (saved) setTheme(saved);
}, []);
