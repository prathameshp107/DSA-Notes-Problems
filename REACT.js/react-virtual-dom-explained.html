Virtual DOM vs Real DOM in React

A complete visual guide to how React's reconciliation engine works — from state change to screen update.

1. What is the Real DOM?

The Real DOM (Document Object Model) is the browser's live, in-memory tree representation of your HTML page. Every HTML tag becomes a node, every node is a JavaScript object, and the browser uses this tree to decide which pixels to paint on screen.

The problem: The Real DOM is expensive to update. Every change — even a single text node — can trigger the browser's full rendering pipeline: style recalculation → layout reflow → repaint. Doing this 10 times for 10 list updates is 10× the cost.

2. What is the Virtual DOM?

The Virtual DOM is React's lightweight JavaScript copy of the Real DOM — a plain object tree that lives entirely in memory. It costs almost nothing to create, and can be thrown away and rebuilt in microseconds.

When state changes, React doesn't touch the Real DOM immediately. Instead it builds a new Virtual DOM, compares it to the previous one (diffing), and figures out the minimum Real DOM changes needed. This process is called reconciliation.

Here's what a Virtual DOM node actually looks like under the hood:

// What JSX compiles to — just a plain JS object
// <div className="card"><h1>Hello</h1></div>
{
  type: "div",
  props: { className: "card" },
  children: [
    {
      type: "h1",
      props: {},
      children: ["Hello"]
    }
  ]
}
Key insight: The Virtual DOM is just a JavaScript object. No browser APIs, no reflows, no repaints. Comparing two plain objects is orders of magnitude cheaper than mutating the Real DOM.

4. The diffing algorithm (reconciliation)

React's diffing has three rules that make it O(n) instead of the theoretical O(n³) of a naive tree comparison:

Rule 1 — Different element types = full subtree replace

If a <div> changes to a <span>, React tears out the entire subtree and rebuilds from scratch. The child component unmounts and remounts — all state is lost.

// Before
<div><Counter /></div>

// After — div → span means Counter unmounts and remounts fresh
<span><Counter /></span>

Rule 2 — Same type = update in place

If the element type stays the same, React only updates the changed attributes/props. The DOM node itself is reused.

// Before
<button className="btn" disabled={false}>Save</button>

// After — React patches only the className attribute
<button className="btn active" disabled={false}>Save</button>
// Real DOM patch: element.className = "btn active"

Rule 3 — Lists need key props

Without keys, React can't tell which list item moved vs. which was deleted. With keys, it tracks identity across renders — moving DOM nodes instead of recreating them.

// Bad — React re-renders everything on sort/reorder
{items.map(item => <li>{item.name}</li>)}

// Good — React moves existing DOM nodes, doesn't recreate them
{items.map(item => <li key={item.id}>{item.name}</li>)}

7. A concrete code example

function PriceTag({ price }) {
  return <span className="price">${price}</span>;
}

function App() {
  const [price, setPrice] = useState(10);

  return (
    <div>
      <h1>Product</h1>               {/* never changes */}
      <PriceTag price={price} />     {/* only this re-renders */}
      <button onClick={() => setPrice(p => p + 1)}>
        Increase price
      </button>
    </div>
  );
}

When you click the button, React builds a new Virtual DOM for the whole <div> — but after diffing, it makes exactly one Real DOM write: updating the text content of the <span>. The <h1> and <button> are never touched in the Real DOM.

8. Why not just use the Real DOM directly?

You could — and before React, jQuery apps did exactly this:

// jQuery style — you manually track everything
$('#price').text(newPrice);
$('#badge').addClass('updated');
$('#count').text(parseInt($('#count').text()) + 1);
The problem: As your app grows, manually tracking which DOM nodes need updating for each state change becomes impossibly complex. UI goes out of sync with data. React flips this: you describe what the UI should look like given the current state, and React figures out the minimum changes to get there.

9. Key takeaways

  • The Real DOM is expensive — every change can trigger style recalculation, layout reflow, and repaint.
  • The Virtual DOM is just a JavaScript object tree — near-zero cost to create and compare.
  • Reconciliation (diffing) finds the minimum diff between old and new Virtual DOM trees.
  • Only the diffed changes get applied to the Real DOM in a single batch.
  • key props are critical for list reconciliation — without them React can't track element identity.
  • Changing an element's type (e.g. div → span) always destroys and remounts the full subtree.
  • React Fiber (the modern engine) can pause and resume reconciliation, prioritising user interactions over background updates.
